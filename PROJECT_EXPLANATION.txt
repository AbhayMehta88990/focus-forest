================================================================================
                        FOCUS FOREST - PROJECT EXPLANATION
================================================================================


1. PROJECT STATEMENT
--------------------
Students often struggle with staying focused during study sessions. There is no
fun or motivating way to track focus time, compete with friends, or study
together in real-time. Focus Forest solves this by turning study sessions into
a gamified experience where you grow virtual 3D trees by staying focused.


2. SOLUTION
-----------
Focus Forest is a full-stack web application that:
- Lets users start Pomodoro-style focus timers (10–180 minutes)
- Grows a 3D plant model in real-time as the user stays focused
- If the user quits early, the plant dies (failed session)
- If they complete the timer, the plant is added to their virtual forest
- Users can view their focus history, analytics dashboard, and leaderboard
- "Study Together" rooms let multiple users join, chat, share camera, and
  focus together in real-time using WebSockets and WebRTC


3. LANGUAGES & TECHNOLOGIES USED
---------------------------------
FRONTEND:
  - JavaScript (React.js) — UI components and logic
  - HTML/CSS (Tailwind CSS) — Styling and layout
  - Three.js (React Three Fiber) — 3D plant model rendering
  - STOMP.js + SockJS — WebSocket client for real-time features
  - WebRTC — Peer-to-peer camera sharing in study rooms

BACKEND:
  - Java (Spring Boot 3) — REST API and business logic
  - Spring Security + OAuth2 — Google authentication
  - Spring WebSocket (STOMP) — Real-time messaging for study rooms
  - Hibernate/JPA — Database ORM
  - MySQL (Aiven Cloud) — Production database
  - Docker — Container deployment on Render

DEPLOYMENT:
  - Frontend: Vercel (auto-deploys from GitHub)
  - Backend: Render (Docker container)
  - Database: Aiven Cloud MySQL


4. IMPORTANT API ENDPOINTS
--------------------------
AUTH:
  GET  /api/auth/me          — Returns current logged-in user info
  POST /api/auth/logout       — Logs out the user

SESSIONS (Focus Timer):
  POST /api/sessions          — Save a completed/failed focus session
  GET  /api/sessions          — Get all sessions for the logged-in user

ANALYTICS (Dashboard):
  GET  /api/analytics         — Get focus stats (total time, trees, daily data)

LEADERBOARD:
  GET  /api/leaderboard       — Get top users ranked by total focus time

STUDY ROOMS:
  POST /api/rooms             — Create a new study room (returns room code)
  GET  /api/rooms/{code}      — Get room info by code

WEBSOCKET (Real-time):
  /ws                         — SockJS/STOMP WebSocket endpoint
  /app/room/{code}/join       — Announce joining a room
  /app/room/{code}/leave      — Announce leaving a room
  /app/room/{code}/chat       — Send a chat message
  /app/room/{code}/timer      — Start/stop shared timer
  /app/room/{code}/signal     — WebRTC signaling (camera sharing)


5. GOOGLE OAUTH INTEGRATION
----------------------------
How it works:
  1. User clicks "Sign in with Google" on the login page
  2. Browser redirects to /oauth2/authorization/google (Spring Security)
  3. Spring redirects to Google's consent screen
  4. User approves → Google redirects back with an auth code
  5. Spring exchanges the code for user info (name, email, picture)
  6. If user is new, a record is created in the database
  7. A session cookie (JSESSIONID) is set for future requests
  8. User is redirected to the frontend

Key files:
  - SecurityConfig.java — Configures OAuth2 login, success/failure handlers
  - application.properties — Stores Google client ID/secret (via env vars)
  - AuthContext.jsx — Frontend checks /api/auth/me to know if user is logged in


6. IMPORTANT CODE EXPLAINED
----------------------------

a) SecurityConfig.java (Backend - Authentication)
   - Configures which URLs need authentication and which are public
   - On successful Google login, saves user to database and redirects to frontend
   - On logout, clears session cookie
   - Sets up CORS to allow frontend to call backend APIs

b) StudyWebSocketController.java (Backend - Real-time)
   - Tracks participants per room in a ConcurrentHashMap (thread-safe)
   - When someone joins, sends the FULL participant list to all users
   - Relays chat messages, timer events, and WebRTC signals between users

c) Timer.jsx (Frontend - Focus Timer)
   - Manages the countdown timer with start/stop/reset functionality
   - Calls POST /api/sessions to save results when timer completes or fails
   - Renders a 3D plant model that grows through 3 stages (seed → plant → tree)
   - Plant type selector lets users choose coffee, tree, palm, or bamboo

d) StudyTogether.jsx (Frontend - Study Rooms)
   - Connects to backend via STOMP/SockJS WebSocket
   - Manages participants list, chat messages, and shared timer
   - Uses WebRTC for peer-to-peer camera sharing between participants
   - When a new user joins, existing users re-announce themselves so everyone
     discovers each other

e) ModelViewer.jsx (Frontend - 3D Models)
   - Uses React Three Fiber (Three.js) to render .glb 3D plant models
   - Each plant type has 3 models: seed (1.glb), growing (2.glb), mature (3.glb)
   - Models auto-rotate and are normalized to fit the viewport

f) AuthContext.jsx (Frontend - Auth State)
   - Creates a React Context that wraps the entire app
   - On page load, calls /api/auth/me to check if user is logged in
   - Provides login() and logout() functions to all components
   - login() redirects to Google OAuth, logout() calls the backend then reloads

g) config.js (Frontend - URL Configuration)
   - In production: API calls use relative URLs (go through Vercel proxy)
   - In development: API calls go directly to localhost:3001
   - WebSocket always connects directly to the backend (Vercel can't proxy WS)


7. DATABASE TABLES
------------------
  users        — id, google_id, email, name, picture
  sessions     — id, user_id, duration, category, outcome, plant_type, started_at
  study_rooms  — id, room_code, host_user_id, timer_duration, status, created_at
